# Прототип защищенной веб-службы (Практика 3)

## Суть практики
Мы реализовали каркас приложения, который ставит во главу угла **безопасность конфигурации** и **защиту периметра**. Основная цель - не допустить запуск службы с дырами в настройках и ограничить доступ внешнего мира.

## Инструкция по запуску

### 1. Установка зависимостей
```
npm install
```
2. Учебный режим (Learning)
Мягкие лимиты, подробные логи ошибок.
```
# Через переменные окружения
APP_MODE=learning PORT=3000 npm start
# Или через аргументы (наивысший приоритет)
node src/app.js --mode=learning --port=3000
```
3. Боевой режим (Production)
Строгие лимиты, скрытие подробностей ошибок, жесткий CORS.
```
node src/app.js --mode=production --port=8080 --origins=[https://my-app.com](https://my-app.com)
```

## Проверки работоспособности

1. Проверка приоритетов:

* Создайте файл .env с `PORT=3000`.
* Запустите `node src/app.js --port=4000`.
* Результат: Приложение запустится на 4000, так как аргументы важнее файла.

2. Проверка валидации:

* Попробуйте запустить `node src/app.js --mode=invalid`
* Результат: Приложение выведет ошибку валидации и завершит процесс (код 1).

3. Проверка Rate Limit:

* Выполните 6 POST-запросов к `/data` в течение минуты.
* Результат: 6-й запрос вернет ошибку `429 Too Many Requests`.

4. Проверка CORS:

* Попробуйте отправить запрос с неописанного в настройках домена.
* Результат: Заголовок Access-Control-Allow-Origin не будет соответствовать источнику, браузер заблокирует ответ.

## Пояснение к настройкам

### Критичные настройки:

* `TRUSTED_ORIGINS`: Без списка доверенных доменов злоумышленник может выполнить атаку CSRF или украсть данные через скрипты на сторонних сайтах.
* `APP_MODE`: От него зависит детализация ошибок. В production ошибки не должны показывать `stack trace`, чтобы не раскрывать структуру кода.
* `PORT`: Ошибка в порту может привести к конфликту служб или невозможности запуска в контейнере.

### Раннее обнаружение ошибок

Реализована проверка до сборки приложения (в блоке `validateConfig`). Это снижает риски полуживого состояния системы, когда сервис запустился, но не может корректно работать из-за кривого конфига. Проще уронить сервис сразу с понятным логом, чем разбираться в причинах падения под нагрузкой.

### Вывод

В ходе работы был создан устойчивый каркас. Я научилась:

1. Иерархическому чтению настроек (File < ENV < CLI).
2. "Fail-fast" подходу — приложение не стартует с опасными или неверными данными.
3. Базовой защите через Helmet (защитные заголовки), CORS (белый список) и Rate Limiting (защита от DDoS).
